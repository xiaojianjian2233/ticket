# 智齿工单接口速查

本文档汇总系统对接智齿客服（Sobot）工单系统**实际调用**的接口，含完整地址、请求参数与返回判断。
依据 `feishu-python/app/zhichi_client.py` 的实际实现整理。

---

## 一、公共前提

### 1.1 地址头（Base URL）

代码中 `_ZHICHI_BASE` 写死为：

```
https://www.soboten.com
```

> ⚠️ 注意：官方文档示例地址为 `https://www.sobot.com`，代码实际用的是 `www.soboten.com`（国内域名）。以代码为准。

### 1.2 鉴权与通用约定

- 除 `get_token` 外，所有业务接口都需在 HTTP header 携带 `token` 参数。
- Token 在 `zhichi_client.py` 中**模块级缓存**，TTL 依据接口返回的 `expires_in`（默认 86400 秒）计算，提前 5 分钟失效，`threading.Lock` 保证线程安全。
- 请求统一走 `_request()`：HTTP 401 或返回体 `ret_code` 为 `100001`/`100002`（或 `ret_msg` 含 token 关键字）时，自动刷新 token 重试一次。
- **业务成功判断**：返回体 `ret_code == "000000"`，否则取 `ret_msg` 报错。
- 业务接口 header：`{"token": token, "content-type": "application/json"}`。

### 1.3 签名算法（get_token 用）

```
create_time = 当前秒级时间戳（字符串）
sign = md5(appid + create_time + app_key)   # 小写 hex
```

`appid` / `app_key` 由智齿售后提供，对应 `.env` 的 `ZHICHI_APPID` / `ZHICHI_APP_KEY`。

## 二、获取访问 Token（get_token）

> 对应代码：`zhichi_client._get_token()`

`GET {base}/api/get_token`

| 参数 | 位置 | 必填 | 说明 |
|------|------|------|------|
| `appid` | query | 是 | 第三方调用唯一凭证ID（`ZHICHI_APPID`） |
| `create_time` | query | 是 | 秒级时间戳字符串 |
| `sign` | query | 是 | `md5(appid + create_time + app_key)` |

**返回（成功）：**
```json
{
  "ret_code": "000000",
  "ret_msg": "操作成功",
  "item": { "token": "4ac37cb2...", "expires_in": "86400" }
}
```

token 在 `item.token`，有效期 `item.expires_in`（秒）。

## 三、查询工单详情（get_ticket_by_id）

> 对应代码：`zhichi_client.get_ticket_by_id()`

`GET {base}/api/ws/5/ticket/get_ticket_by_id`

| 参数 | 位置 | 必填 | 说明 |
|------|------|------|------|
| `ticketid` | query | 是 | 智齿工单ID |

返回工单对象在 `item` 下（含标题、内容、状态、优先级、坐席等）。`ret_code != "000000"` 视为失败。

---

## 四、查询数据字典（get_data_dict）

> 对应代码：`zhichi_client.get_agent_by_name()`，用于按坐席名查 `agentid`。坐席列表本地缓存 30 分钟。

`GET {base}/api/ws/5/ticket/get_data_dict`

请求参数：无。

**返回 `item` 关键集合：**

| 字段 | 说明 |
|------|------|
| `ticket_type_list` | 工单分类（多级树，动态） |
| `ticket_status_list` | 工单状态枚举 |
| `ticket_level_list` | 工单优先级枚举 |
| `group_list` | 技能组（`groupid` / `group_name`） |
| `agent_list` | 坐席列表（`agentid` / `agent_name`） |

代码逻辑：遍历 `item.agent_list`，按 `agent_name == 传入姓名` 匹配，返回 `{agentid, agent_name}`，未找到返回 `None`。

---

## 五、上传附件（upload_file）

> 对应代码：`zhichi_client.upload_file()`。回复工单带附件时先上传得到 `file_url`。

`POST {base}/api/ws/5/ticket/upload_file`

- header：仅 `token`（不设 content-type，由 requests 按 multipart 自动设置）
- body：`multipart/form-data`

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `file` | file | 是 | 文件内容，形如 `(file_name, file_content)` |
| `file_num_key` | form | 是 | 传文件名 |

返回 `item.file_url`，`ret_code != "000000"` 视为失败。

---

## 六、坐席回复工单（save_ticket_reply）

> 对应代码：`zhichi_client.reply_ticket()`，触发入口 `/webhook/zhichi/replyOrder`。

`POST {base}/api/ws/5/ticket/save_ticket_reply`

| 参数 | 必填 | 说明 |
|------|------|------|
| `ticketid` | 是 | 工单ID |
| `ticket_title` | 是 | 工单标题 |
| `ticket_content` | 是 | 工单问题描述 |
| `get_ticket_datetime` | 是 | 获取工单信息时间，代码取当前时间 `YYYY-MM-DD HH:MM:SS` |
| `reply_content` | 否 | 回复内容（即解决方案说明） |
| `reply_type` | 是 | `0` 所有人可见（默认）/ `1` 仅坐席可见 |
| `reply_agentid` | 是 | 回复坐席ID（由坐席名经数据字典查得） |
| `reply_agent_name` | 是 | 回复坐席名称 |
| `ticket_status` | 是 | 工单状态（见枚举），不传则保持当前 |
| `ticket_level` | 是 | 工单优先级（见枚举） |
| `reply_file_str` | 否 | 回复附件路径，多个用英文分号 `;` 隔开 |

> 结构化拼接（业务层 handler 处理）：`productCategory`/`productModule`/`rootCause` 任一有值时，`reply_content` 拼为 `产品大类：xx<br>产品问题模块：xx<br>分析根因：xx<br>解决方案说明：{replyContent}`（智齿用 `<br>` 换行）。

**返回（成功）：**
```json
{ "ret_code": "000000", "ret_msg": "操作成功" }
```

---

## 七、调用链（replyOrder 全流程）

```
本地缓存 data/zhichi_tickets/{ticketid}.json 读取工单数据
→ get_data_dict 查坐席 agentid（按 handleUser 姓名匹配）
→ （如有附件）upload_file 得到 file_url，拼 reply_file_str
→ save_ticket_reply 回复工单
→ 更新本地缓存并回写飞书
```

依赖 ZhichiTicketStore 中有对应 ticketId 的记录（工单需推送过 `/webhook/zhichi`）。

## 八、枚举值

### 工单状态（ticket_status）
| 值 | 含义 |
|----|------|
| 0 | 尚未受理 |
| 1 | 受理中 |
| 2 | 等待回复 |
| 3 | 已解决 |
| 98 | 已删除 |
| 99 | 已关闭 |

### 工单优先级（ticket_level）
| 值 | 含义 |
|----|------|
| 0 | 低 |
| 1 | 中 |
| 2 | 高 |
| 3 | 紧急 |
