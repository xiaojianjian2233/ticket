## 〇、全局前端约定

### 数据状态标签色板（6 态，全流程复用）
| status | 标签 | 色 |
|---|---|---|
| pending | 待处理 | 灰 |
| returned | 已退回(不接管) | 红 |
| pending_manual | 待人工 | 橙 |
| pending_rd | 待研发 | 蓝 |
| done | 已处理 | 绿 |
| closed | 已关闭 | 深灰 |

### 前端数据边界（重要）
- ✏️**前端只展示 t_ticket_info 归一字段**；`t_ticket_org`(原始结构化) + `t_ticket_detail.raw_json`(全量报文) **只存库、不上前端**（PII 在 detail 也不直接列表展示，从源头规避脱敏复杂度）。
- 来源原始数据仅供后端回写/排障，不暴露页面。

### 权限基线
- admin：全部读写 + 运维 + 配置 + 原始数据（后台）。
- handler：业务读写（转人工/关闭/改责任人）。
- visitor：只读 + 只读统计。

## 流程 01 · 智齿/KSM 工单接入、参数校验、重复过滤 ✅

### 后端

**入站（接入层，只入队不处理）**
- KSM `POST /webhook/ksm`：校验 webhook 鉴权 → 提取 noticeNum → 入队 `ksm_intake` → 立即 200。（不在请求线程拉全量，subscribeCallback 慢 60s）
- 智齿 `POST /webhook/zhichi`：校验鉴权 → 入队 `sync_ticket{raw}` → 200。
- assistant：对话提单 → 入队 run_pipeline（见流程09）。

**worker 拉全量 + 归一**
- ksm_intake：`subscribeCallback(noticeNum,60s)` 拉全量；status≠true/超时 → 重试≤3 → abandoned(自动 pending_manual+通知)。
- normalize：HTML清洗 + 时间戳(ms→timestamptz) + 附件→MinIO(KSM 需 UA 头)→t_attachment+public_url + 字段映射到 `t_ticket_org`(结构化)/`raw_json`(全量)/`t_ticket_info+detail`(归一)；has_attachment 回填。

**参数校验**
- Pydantic：报文结构/必填(noticeNum/ticketid)/类型。
- 业务：source 合法、title/description 非空、时间戳合法。
- 失败 → 记 integration_log(fail)+告警，**不入库**（脏数据不进流水线）。

**重复/退回过滤（FR-01 幂等核心）**
```
幂等键 (source, source_id)
 ├ 已存在 → 退回单：更新原单(不变号)+return_count+1+is_returned+退回标签
 │         → 照常入队 run_pipeline(跑AI作参考) → 末端强制 pending_manual → 飞书@"第N次退回"
 └ 新编号 → 插入 info(pending)+detail+org → SLA-1启动(工作日+48h) → 入队 run_pipeline
```

**数据流转**：webhook → queue(intake) → worker[org+raw_json+info+detail+attachment] → queue(run_pipeline)。

**异常兜底**
| 异常 | 处置 |
|---|---|
| subscribeCallback 失败/超时 | 重试≤3→abandoned→自动 pending_manual+告警 |
| 附件下载失败 | 不阻断，t_attachment 记 failed 可重试 |
| 报文校验失败 | 记 integration_log+告警，不入库 |
| 并发唯一键冲突 | 退回逻辑接管（幂等） |

**日志**：subscribeCallback→integration_log(异步)；归一每步→skill_log + 结构化日志(trace_id 贯穿)。

### 前端

- **菜单/路由**：工单管理→`/tickets`(列表)、`/tickets/:id`(详情)。
- **列表列**：工单号/来源(标签)/标题/产品线/功能模块/责任人(空=未分配橙)/状态(色板)/退回(×N红标)/创建时间/SLA(超时红)/操作。
- **筛选**：来源/状态/产品线/责任人/是否退回/时间范围 + 关键词(ticket_no/title)。
- **详情页**：基本信息 + 附件(MinIO缩略图) + 退回黄条(第N次退回)。✏️**不展示 org/raw_json/PII**，只 info 字段。
- **运维按钮**（✏️新增，admin）：**"手动重新拉取/重新入队"**——处理 abandoned/拉取失败工单，重新触发 intake/pipeline。
- **交互**：列表骨架屏；加载失败重试；pending 显"AI处理中"转圈；webhook 自动接入，无手动接入入口（仅运维重入队按钮）。
- **按钮权限**：查看(全部)；转人工/关闭/改责任人/重新入队(admin/handler，重入队限 admin)；导出(admin/handler)。

## 流程 02 · 工单流转判断、不可流转处理 ✅

### 后端

**S1 ticket-routable（pipeline 第一步，SKILL.md 可热改）**
- 喂入：title/description/product_name/module/has_attachment + 关键词/模块清单(SKILL.md 内)。
- 按序判定(命中即停)：FR-02 描述<15字且无附件且LLM判语义不明→不可流转；FR-03 多问题(≥2编号/分隔词接≥10字)→多问题；FR-04 非发票云模块黑名单且无关键词→不可流转；FR-05 完全无发票云关键词→不可流转；FR-06 否则→可流转。

**分支**
- 不接管(按来源不对称)：KSM=飞书通知+status=returned(零回写)；智齿=save_ticket_reply(固定话术+ticket_status=3)+通知；assistant=飞书私信。写 route_action/route_reason，结束。
- 可流转：KSM lockKsmOrder 接管 → 进 S2 打标。
- 退回单：照常跑判断，不接管按上述/可流转则续跑末端强制 pending_manual。

**异常兜底**
| 异常 | 处置 |
|---|---|
| Claude 失败 | 降级 DeepSeek |
| 两者皆失败 | 任务挂起等待（不猜测不误放） |
| lockKsmOrder 失败 | 重试≤3→告警人工接管 |
| save_ticket_reply 失败 | 重试≤3→writeback_error |

**日志**：S1→skill_log(route_action/reason/evidence/model_used)；lock/reply→integration_log。

### 前端

- **状态**：不接管→returned(红)；可流转→随流水线推进。
- **✏️未接管工单单独视图**：菜单"工单管理→未接管工单"，独立 tab/路由 `/tickets/unhandled`，筛 status=returned。
- **详情页流转区块**：流转结果(可流转绿/不接管红) + route_action + route_reason + 来源动作说明；✏️**判定依据 evidence 对 admin+handler 可见**(人工复核 AI)。
- **改判按钮**（✏️未接管列表，admin）："手动改判可流转 + 重新入队"——AI 误判时纠正重跑。✏️**机器人对话改判归后续规划**（第9条本期助手只读，写操作统一后续放开）。
- **交互**：流转判定中显"流转判断中"转圈；不接管详情红 banner+原因(无后续按钮)；LLM 挂起显"等待AI服务恢复"灰 badge。
- **按钮权限**：查看 evidence(admin/handler)；手动改判/重入队(admin)。

## 流程 03 · Skill 编写 / 重复合并 / 自动打标 ✅

### 后端

**Skill 编写与热加载（modules/skill）**
- admin 编辑 SKILL.md → `PUT /api/v1/skills/{name}` → 校验 frontmatter+正文 → 存 t_skill_md + history + operation_log(前后值) → bump version + 清缓存 → 热生效。
- runtime：skill_runner 从 t_skill_md 读(缓存+version失效)喂 LLM。9 个可编辑 SKILL。

**S3 info-dedup 短路复用（打标后→FAQ前，第2条）**
- 硅基流动 top8(同产品线近120天) → 阈值0.85 + LLM二次确认。
- 命中(有效答复最近单)→短路复用 final_reply→writeback关单→resolved_at→结束；命中无有效答复→不短路续跑；未命中→进FAQ。
- 记 dup_ticket_ids + reused_from_ticket_id + t_ticket_merge(dedup_reuse/dedup_marked)。

**S2 ticket-tagging 打标（第3条）**
- 从 t_module_owner 读候选(14产品线+157模块+触发词)喂 LLM → 选产品线(单)+模块(单)[责任人不进prompt] → harness 按 func_module 查 t_module_owner 得 dev_owner。
- 写 info(冗余当前值) + t_ticket_tag(明细 is_current=true, confidence/tag_source/model)。
- 兜底：产品线/模块无法判断→标"无法判断"续跑；责任人查表无→dev_owner_missing+默认接口人。

**异常兜底**
| 异常 | 处置 |
|---|---|
| 硅基流动挂(去重) | 降级"非重复"续跑 |
| LLM 打标失败 | Claude→DeepSeek；都失败标"无法判断"不阻断 |
| dev_owner 查表无 | dev_owner_missing + 默认接口人 |
| SKILL.md 校验失败 | 拒绝保存+提示 |

**日志**：SKILL 执行→skill_log；SKILL.md 编辑→operation_log(前后值+版本)；打标→t_ticket_tag(重打标旧行 is_current=false)。

### 前端

- **Skill 管理页**（admin，`/skills`）：列表(name/类型/version/更新人/时间)；9 LLM 类→Markdown 编辑器，py 类→只读"逻辑在代码"；✏️**样例试跑预览**(改后用样例工单试跑看效果不落库)；版本历史 diff + 回滚。
- **打标区块**（工单详情）：当前标签(产品线/模块/责任人，空=未分配橙) + 置信度(✏️confidence 可空，有则显示) + 判定来源标签(AI/人工/兜底) + 打标历史时间线(is_current 高亮)；**人工修正**(handler 改→插 manual 行+刷主表+确认弹窗，候选从 t_module_owner 取)。
- **去重区块**（工单详情）：重复标记"重复(复用历史答复)" + 复用来源链接(reused_from_ticket_id) + 相似度 + 短路提示"复用 #xxx 答复自动关单"。✏️**不做撤销**(复用即终态)。
- **交互**：SKILL.md 保存校验失败 toast/成功"已热生效"；打标中转圈；改标签下拉(候选 t_module_owner)+确认弹窗；责任人未分配橙提示可手动指派。

## 流程 04 · FAQ 检索 + Agent 答复四分支 ✅（分类逻辑待 agent 示例细化）

### 后端

**S4 faq-retrieve（info-dedup 未命中后）**
- desensitize(query) → 硅基流动 embedding 比对 t_faq(同产品线,过滤rejected) 取最高分。
- >0.8 命中→final_reply=FAQ正文→writeback关单→resolved_at→hit_count+1→结束；未命中/失败(重试3次)→进 S6。

**S6 ask agent（open-api-channel 四步异步）**
- desensitize(question)→get_token→ask_init(动态cid)→answer_async(question+images[public_url]+uid)→轮询60s×3→end_session。
- 返回 answer+transfer_result；ERROR/超时/100002/空/TRANSFER→转C。

**S7 answer-router（SKILL 语义判定，🟡待示例细化）**
- 先看 transfer_result：TRANSFER→C；NO_ACTION→LLM语义分类：bug→A/需求→B/正确回答需补信息→C/正确回答已给方案→D；无法识别→兜底C。

**四分支**
- **A bug**：hub-dedup(0.7+LLM)→重复关联rd_hub_id/新建hub+linear-sync(CNPRD,默认in_progress)→不回写客户,pending_rd,SLA-2(hub创建起)启动。
- **B 需求**：dispatch派单→pending_manual→飞书群@处理人(full_reply)。
- **C 补料**：dispatch派单→pending_manual→KSM supplyKsmOrder/智齿 ticket_status=2/assistant私信"需要您提供…"。
- **D 正常**：提取两段→reply-humanize→desensitize→final_reply→writeback关单(KSM处理完成4/智齿3/assistant私信)→resolved_at→进FAQ收录(流程05)。

**异常兜底**
| 异常 | 处置 |
|---|---|
| 硅基流动检索失败 | 重试3次→视未命中走agent |
| agent 失败/超时/TRANSFER | 转C |
| reply-humanize 失败 | 原文回写不阻断 |
| hub-dedup 硅基挂 | 降级"非重复"新建hub |
| linear-sync 失败 | hub保留+重试≤3→告警人工 |
| 回写失败 | writeback_error+重试≤3 |

**日志**：各步→skill_log；agent/linear/回写→integration_log。

### 前端

- **答复区块**（工单详情）：答复来源标签(FAQ命中/AI答复/复用历史) + 分支 A/B/C/D 色标 + final_reply + transfer_result；✏️**admin 可对比 ai_reply/humanized_reply**(监控去AI化质量)。
- **分支差异展示**：A=已转研发+hub链接+Linear链接+pending_rd蓝；B=已转人工(需求)+派单人+橙；C=等待补料+引导文案+橙；D=已答复关单+done绿+收录FAQ状态。
- **✏️handler"我的待办"工作台**（`/workbench`，B/C 派到 dispatch_assignee=当前用户）：显示 full_reply(AI参考)；✏️**操作多路径**——确认AI答复关单(有AI答复时一键确认回写)/修改后回写/转研发(转A)/补充资料；处理后状态流转。
- **交互**：agent 处理中"AI答复生成中(可能需1-3分钟)"转圈；FAQ命中"命中知识库自动答复"绿提示+链接；分支对应色标文案；去AI化对比折叠区(admin)。

## 流程 05 · FAQ 自动收录 / DeepSeek 审核 / 异常通知 ✅

### 后端

**S10 faq-record（D 答复后收录）**
- 收录前去重：硅基流动 0.85 与同产品线 t_faq 比对，≥0.85→no_faq 终止。
- 归纳：用 humanize+脱敏后 description+final_reply → {title≤20,content≤300} → 算 embedding → 写 t_faq(pending_review) → 触发 S11。

**S11 faq-review（先入库后审，不阻断）**
- DeepSeek 四维审核(①敏感②事实③内部④质量)→写 t_faq_review(result/四维bool/reject_dims/reason)。
- 四维全过→approved；任一不过→rejected(检索排除软下线)+飞书@审核人(notified防重发)。

**异常兜底**
| 异常 | 处置 |
|---|---|
| 收录去重硅基挂 | 降级"不重复"照常收录 |
| faq-record LLM失败 | Claude→DeepSeek；都失败跳过收录(不阻断关单) |
| faq-review 失败 | 保持 pending_review(可被检索)，记failed重试 |
| 飞书通知失败 | 重试≤3，notified 标记 |

**日志**：faq-record/review→skill_log；审核→t_faq_review(每次一行)；通知→integration_log。

### 前端

- **FAQ 管理页**（`/faq`）：列表(faq_no/title/产品线/审核状态标签/命中次数/来源工单/时间)；筛产品线/状态/关键词；操作 查看/编辑/删除(admin)/审核历史。
- **FAQ 审核页**（`/faq/review`，审核人）：✏️显示 pending_review+rejected(rejected红高亮置顶)；审核详情 title/content+四维结果(✓/✗)+reject_dims+reason；✏️**人工操作 通过/驳回/编辑/删除(人工最终决定权,覆盖AI)**；审核历史时间线(t_faq_review)。
- **状态标签**：pending_review(待审灰)/approved(通过绿)/rejected(驳回红，检索排除)。
- **编辑表单**：title(≤20校验)/content(≤300校验)/product_tag(下拉)/图片(MinIO)；✏️**人工编辑后直接 approved，不重新 AI 审核**。
- **删除**：✏️**物理删**（FAQ 破例，区别于"线上只加不删"）；连带删 t_faq_review 记录；确认弹窗"驳回/删除后不再被检索"。
- **交互**：rejected 飞书卡片(标题/工单/驳回维度/reason+审核页链接)；驳回确认弹窗。

## 流程 07 · 飞书 SSO 登录集成 ✅

> 照搬 jfservice-auto/auth.py，适配 RBAC 三级。纯 SSO 无本地账密。

### 后端

**登录流程（OAuth 授权码）**
- `GET /auth/feishu/login` → 返回飞书授权 URL(app_id+redirect_uri+response_type=code)。
- `GET /auth/feishu/callback?code` → app_access_token → user_access_token(oidc) → user_info → 同步 t_users(按 feishu_uid 查→按姓名查导入记录→新建 default visitor) → 禁用拒登(403) → 签发 JWT(sub/name/role) → 返回 token+user。
- `GET /auth/me`(带JWT) → 当前用户信息。

**鉴权中间件**：Bearer JWT → decode → 注入 current_user → require_role 校验；无/过期→401；角色不足→403。

**异常**：飞书 token/user_info 失败→502；账号禁用→403；JWT 过期→401跳登录。
**日志**：登录成功/失败→结构化日志+last_login_at；OAuth 调用→integration_log。
**JWT 有效期**：✏️**8 小时**（一个工作日，`JWT_TTL_SECONDS`，过期重登）。

### 前端

- **登录页**（`/login`）：Logo+系统名+"飞书登录"按钮；✏️**纯 SSO 无账密框**；点登录→飞书授权→回调 code→换 JWT→存→跳首页。
- **鉴权交互**：未登录→重定向 /login；401→清 token+跳登录+"登录已过期"；403→无权限提示；登录中 loading；账号禁用提示。
- **全局用户态**：顶部显 name/avatar/role；✏️**菜单按 role 动态渲染**；退出=清 JWT+跳登录。
- **✏️新用户默认 visitor**（首登自动建号只读，需 admin 提权）。
- **✏️菜单可见性**：visitor=工单查看/知识库查阅/统计；handler=+工作台/处理；admin=+配置(Skill/派单/用户)/管理。
- **路由守卫**：无 token→/login；admin 路由(Skill管理/派单配置/用户管理)非 admin 拦截。

## 流程 08 · SLA 定时任务、超时判定、通报 ✅

### 后端

**sla_scanner（scheduler 进程，每日北京 08:00）**
- 节假日校验(holiday_service 国务院API)→workday 算工作日到期。
- SLA-1 人工(info)：扫 status未闭环 AND now>sla_due_at → 按 ai_dev_owner 计数+单号(dev_owner_missing→"未分配") → 人工SLA群。
- SLA-2 研发(hub)：扫 now>rd_sla_due_at → 按 hub.dev_owner 计数+单号 → 研发SLA群(不同webhook)。
- 每条超时→t_sla_log(去重留痕)。

**计时停表**：SLA-1 起点=created_at 停于 resolved_at；SLA-2 起点=hub创建 停于 rd_resolved_at(Linear回调)；观察期不跑；全程不暂停；bug单转研发后停SLA-1由SLA-2接力。

**observe_scanner**：resolved_at+14天且未退回→本地closed(不回写)。

**queue_monitor**：积压>N/最老>M分→系统告警(独立群)；依赖全挂→挂起探活；abandoned→告警+自动pending_manual+手动重入队。

**异常**：节假日API挂→本地t_holiday兜底；通报失败重试≤3；重入→sla_notified_marks按"日期+08:00"去重。
**日志**：扫描→结构化日志；超时→t_sla_log；通报→integration_log。

### 前端

- **SLA 监控页**（`/sla`，✏️全部人员可看全部）：✏️**合并列表用 sla_type 区分**(人工/研发)；概览卡片(超时总数/今日新增/责任人top)；列表(工单号/责任人/超时时长/SLA类型/状态/超时天数)；筛选(类型/责任人/产品线/超时区间)；状态标签 正常绿/超时breached红。
- **✏️系统告警同页一个 tab**：队列积压/依赖挂起/abandoned 告警记录 + 通报历史(t_sla_log) + 责任人超时排名。
- **工单详情 SLA 区块**：SLA-1 起止/到期/已超时长/normal-breached；(bug)SLA-2 研发起止；breached 红+已超N工作日。
- **交互**：✏️超时项红行/badge；✏️**简单显到期时间+是否超时**(不做前端工作日倒计时)；通报记录时间线。

## 流程 09 · 智能助手（对话提单 / 数据统计）✅

> 本期只读 NL2SQL + 对话提单；写操作(关闭/改责任人)降为后续规划。

### 后端

**对话提单(submit)**：校验(非空/≥15字/防重复:同人+相同内容+5分钟)→建单 source=assistant+submitter_uid→入队 run_pipeline(完整流水线+计SLA)→记 t_assistant_log(submit)→答复飞书私信提单人。

**数据统计(只读 NL2SQL)**：nl2sql_service 调 assistant-nl2sql SKILL(NL→SQL,可热改)→★sql_guard(py不可热改:强制SELECT-only+LIMIT封顶+PII按角色脱敏)→执行→对话返回(表格/图表)→记 t_assistant_log(query/generated_sql/guard/rows/pii_masked)。

**数据操作**：✏️本期不做(写降后续,统一放开时白名单+二次确认)。

**异常**：NL2SQL失败→"换种问法"；guard拦截→拒绝+记 guard_pass=false+"仅支持查询"；SQL错误→友好提示记 error_msg；提单校验失败→提示原因。
**日志**：每对话→t_assistant_log(90天保留)。

### 前端

- **智能助手面板**（`/assistant`，三级都可用）：对话式聊天界面；引导语"统计数据/提交工单"。
- **对话交互**：统计→生成SQL→✏️**表格为主+简单图表**(数量分布自动柱/饼，复杂后续)；✏️**生成SQL仅admin可见**(审计)，普通用户只看结果；PII visitor脱敏(138****5678)/handler·admin明文；✏️**对话提单确认卡片**(预览标题/描述再建单)→回执"工单已创建#xxx"；多轮 session_id 归组。
- **✏️写操作降级**：收到"关闭/改责任人"等写指令→友好提示"该操作暂未开放，请到工单页操作"。
- **交互细节**：AI思考"正在分析…"；SQL拦截"仅支持查询不支持修改"；无法理解"换种问法/举例"；结果过多"仅显示前N条"。

## 流程 10 · 与 Linear 交互（出站建单 + 回调分派）✅

> 术语：**转研发 = 转 Linear**（同义）。创建 hub 单时即推送 Linear（issueCreate）。

### 后端

**出站：创建 Linear issue（bug 分支 A，S9 linear-sync）**
- A bug → hub-dedup(0.7+LLM)：重复→关联已有 rd_hub_id；新建 hub → **立即 linear-sync issueCreate**(team=CNPRD, 描述=【问题】+full_reply) → 落 linear_issue_id/url, status=in_progress, SLA-2(hub创建起)启动。
- hub 与 Linear issue 同时诞生。

**入站：Linear 回调分派（`POST /webhook/linear`，回写三字段：status/处理人/说明）**

✏️**按 status 精确分派**：
| status | 说明字段 | 动作 |
|---|---|---|
| **计划** | 含发版日期(LLM提取) | **自动答复关单**：话术"您好，您提的问题会在大约{日期}可以修复，公有云环境自动发版、私有化客户需要申请私包发版，如有问题可以咨询发票云在线客服。"→回写**所有关联 info 工单**的客户+关单(✏️固定模板不脱敏) |
| **计划** | 无日期 | **转回 Linear**(同一更新接口，转对应负责人)，附加"未有具体发版日期" |
| **产研退回** | 非空 | 飞书通知 → **转人工** |
| **产研退回** | 空 | **转回 Linear**(转负责人)，附加"未有退回原因" |
| **其它值** | — | hub 表**原样显示**(传什么显示什么)，不处理 |

- 回调更新 hub(rd_handler/status/rd_status_note/rd_callback_at)；status=resolved类(计划+日期自动关单)→填 rd_resolved_at 停 SLA-2；**镜像同步所有关联 info**(rd_status/rd_handler/rd_status_note)。
- ✏️**A 分支终态改为回调驱动自动分派**(不再默认人工关单)。

**出站更新 Linear（退回/转回，同一接口）**：issueUpdate(转 assignee)+commentCreate(附加原因)。契约联调补。

**异常**：issueCreate失败→hub保留+failed+重试≤3→告警；回调验签失败→401；日期提取失败→当无日期走转回；镜像同步info失败→重试(最终一致)。
**日志**：issueCreate/回调/转回→integration_log；hub状态变更→结构化日志。

### 前端

- **研发单管理页**（`/hubs`，admin/handler 可见）：列表(hub_no/title/产品线/责任人/关联工单数N/Linear状态/研发状态/SLA-2/时间)；Linear状态(同步中/已同步链接/失败重试按钮)；研发状态(in_progress蓝/resolved绿/closed灰/其它原样)；操作 查看/打开Linear/(失败)重新同步。
- **研发单详情**：基本信息 + Linear信息(issue_id/链接/同步状态) + 回调信息(rd_handler/rd_status_note/rd_callback_at) + 关联工单(N个info可跳转) + SLA-2。
- **工单详情研发进展区块**(hub→info镜像)：rd_status/rd_handler/rd_status_note + Linear链接 + 计划日期自动答复后显"已答复发版计划"。
- **交互**：✏️**研发状态由 Linear 主动推送，入库即更新**(回调驱动刷新，不轮询)；Linear同步失败红标+重新同步(admin)；自动答复后关联工单显"已按发版计划答复关单"。
