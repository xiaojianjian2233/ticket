# ticket-hub 目录结构设计（落地版）

> 编制：资深后端架构师 · 日期 2026-06-10
> 技术栈：Python + FastAPI + SQLAlchemy(async) + PostgreSQL(pgvector) + MinIO。
> 组织方式：**modular monolith**（按业务模块分包，高内聚，可演进微服务）。
> 进程：`main.py`(API) / `worker.py`(队列消费) / `scheduler.py`(定时)，共用 app 包。

## 二、核心文件详细职责

### 进程入口
| 文件 | 职责 |
|---|---|
| `main.py` | 装配所有 module router、core 中间件、启动/关闭钩子；只跑 API + webhook |
| `worker.py` | 循环消费 `t_task_queue`（ksm_intake/run_pipeline/writeback）；SKIP LOCKED 抢锁；依赖全挂时挂起 |
| `scheduler.py` | 单副本；注册 sla_scanner(每日08:00)/observe_scanner/queue_monitor/holiday_service |

### 横切（core）
| 文件 | 职责 |
|---|---|
| `response.py` | `success(data)`/`fail(code,msg)` → 统一 `{code,message,data}` |
| `exceptions.py` | 异常类型 + FastAPI 全局 handler（转标准返回体，不泄露堆栈，记日志） |
| `security.py` | `issue_jwt`/`decode_jwt`/`get_current_user`/`require_role(admin)`；PII 角色脱敏依赖 |
| `logging.py` | 结构化日志 + trace_id 贯穿 webhook→流水线 |

### 流水线（pipeline）
| 文件 | 职责 |
|---|---|
| `runner.py` | harness 核心：`_run_step(name,fn,payload)` 统一写 t_skill_log；串 S1 routable→S2 tagging→S3 info-dedup(短路)→S4 faq→S6 agent→S6.5 humanize→S7 router→A/B/C/D；退回单末端强制转人工 |
| `steps.py` | 每步实现：调 ai.skill_runner / dispatch.service / knowledge.service / writeback |

### 集成（integrations）
| 文件 | 职责 |
|---|---|
| `base.py` | httpx 基类：超时(HTTP30s/LLM60s)、重试≤3退避、token 缓存、降级钩子 |
| `agent_client.py` | 四步异步：get_token→ask_init(动态cid)→answer_async→轮询60s×3→end_session；transfer_result 透出 |
| `ksm_client.py` | 三步鉴权 + subscribeCallback(60s)/lockKsmOrder/handleKsmOrder(处理完成4)/supplyKsmOrder |

### 关键业务文件
| 文件 | 职责 |
|---|---|
| `ai/skill_runner.py` | 从 t_skill_md 读 SKILL.md(缓存+version失效)喂 LLM；信封出参；不连 DB |
| `assistant/sql_guard.py` | **安全护栏(不可热改)**：强制 SELECT-only、LIMIT 封顶、PII 列按 role 脱敏 |
| `dispatch/service.py` | 按 alloc_value 配额比例分派(基于 t_dispatch_log 统计已分配数)；兜底:溢出组(配比)→默认人→群发 |
| `sla/workday.py` | 工作日累加(排除周末+t_holiday)；双 SLA 到期计算 |
| `intake/idempotency.py` | (source,source_id)已存在=退回→更新原单+return_count+1+退回标签 |

## 三、依赖方向（解耦约束）

```
router → service → repository → db          (模块内纵向)
service → pipeline / ai / integrations       (业务调编排/AI/集成)
ai(skill) ⊥ db                               (Skill 绝不连 DB，信封进出)
integrations ⊥ modules                       (client 不反向依赖业务)
core/common ← 所有层                          (横切被依赖，不依赖业务)
```

- **禁止**：modules 之间横向直接 import service（如 ticket 直接调 knowledge.service）→ 走 pipeline 编排或事件，避免耦合成网。
- **允许**：所有模块依赖 integrations / core / common / db。

## 四、模块 → 业务规则覆盖

| 模块 | 覆盖规则 |
|---|---|
| intake | 第1条 FR-01 退回识别 + 接入 |
| ai + pipeline | 第1条流转/第3条打标/第5条分类/reply-humanize |
| ai/skills + knowledge | 第2条 info-dedup短路/第4条FAQ检索/第6条收录审核 |
| dispatch | 第10条派单 |
| sla | 第7条双SLA + 第8条队列健康 |
| assistant | 第9条只读NL2SQL+提单 |
| feishu | SSO + 统一通知 |
| storage | 附件MinIO + public_url |
| skill | 9 SKILL.md 热加载/编辑 |
