# ticket-hub 系统技术架构设计 v1

> 编制：资深后端架构师 · 日期 2026-06-10
> 依据：《业务规则文档》（9 类规则）+《需求基线 v2》+《Skill 详规 v2》。
> 强制约束：**数据库 PostgreSQL · 文件存储 MinIO · 后端 Python**。
> 参考实现：`jfservice-auto`（已验证技术栈，仅借鉴模式）。

---

## 一、技术栈选型

| 层 | 选型 | 版本/说明 |
|---|---|---|
| **后端框架** | FastAPI + Uvicorn | 0.115 / 异步原生、Pydantic 校验、OpenAPI 自带；与 worker 共用 app |
| **ORM/驱动** | SQLAlchemy 2.0 (async) + asyncpg | 全异步，配合 FastAPI；PG 方言 |
| **数据库** | **PostgreSQL 16** | ★强制。含 `pgvector` 扩展存 FAQ/去重 embedding |
| **向量检索** | pgvector（PG 扩展） | FAQ 检索 / info-dedup / hub-dedup 的余弦相似度在 PG 内算，无需独立向量库 |
| **文件存储** | **MinIO** | ★强制。SDK `minio==7.2`；对外出口 `https://fpy-jfsv.kingdee.com:8864/` |
| **任务队列** | **PG 表队列**（`t_task_queue`）+ 独立 worker 进程 | 不引 Redis/Celery；用 PG `SELECT ... FOR UPDATE SKIP LOCKED` 抢锁，轻量够用 |
| **定时任务** | APScheduler（进程内）或独立 cron 容器 | SLA 扫描(每日08:00)、观察期扫描、队列健康监控 |
| **登录** | 飞书 SSO（OAuth 授权码）+ JWT | python-jose；照搬 jfservice-auto/auth.py |
| **AI - 判断/生成** | Claude → DeepSeek 降级 | httpx 调用，统一 LLM 网关封装 |
| **AI - 向量** | 硅基流动 embedding（bge-m3） | httpx；写入 pgvector |
| **AI - 答复** | ask agent（open-api-channel 四步异步） | httpx；每工单一会话 |
| **AI - 审核** | DeepSeek | faq-review |
| **知识库** | 本地 `t_faq` + pgvector（✏️无 Dify） | 检索主库本地化 |
| **配置** | pydantic-settings + `.env`（密钥走环境变量） | — |
| **HTTP 客户端** | httpx (async) | 统一超时/重试封装 |
| **部署** | Docker Compose（backend + worker + scheduler + PG + MinIO） | 内网部署 |
| **前端** | （沿用 jfservice-auto 前端栈，本文不展开） | — |

> **关键决策：不引入 Redis/Celery/独立向量库**。PG 一库承载业务数据 + 队列 + 向量，降低内网部署运维复杂度。积压量级若未来超 PG 队列承载力，再引 Redis。

## 二、分层架构

```
┌─────────────────────────────────────────────────────────────┐
│ ① 接入层 Inbound/API                                          │
│   webhook(KSM/智齿) · REST API(工单/FAQ/助手/Skill管理) · SSO  │
├─────────────────────────────────────────────────────────────┤
│ ② 业务层 Service（编排，唯一读写 DB）                          │
│   pipeline_runner(harness) · ticket_service · faq_service     │
│   hub_service · assistant_service · writeback_service         │
├──────────────┬────────────────────┬───────────────────────────┤
│ ③ AI服务层    │ ④ 第三方集成层       │ ⑥ 定时任务层               │
│  Skill Runner │  KSM/智齿/ask agent  │  SLA扫描 · 观察期扫描       │
│  (9 SKILL +   │  /硅基流动/DeepSeek  │  队列健康 · 节假日同步      │
│   py skills)  │  /飞书/Linear/MinIO  │                           │
├──────────────┴────────────────────┴───────────────────────────┤
│ ⑤ 数据层 Repository + PG(pgvector) + MinIO                     │
└─────────────────────────────────────────────────────────────┘
        ▲ 业务层经 t_task_queue 解耦 worker（异步流水线）
```

### ① 接入层（Inbound / API）
- **职责**：协议适配、鉴权、参数校验、快速响应（webhook 只入队不处理）。
- **组件**：
  - `webhook`：`/webhook/ksm`（存 noticeNum→入队 ksm_intake）、`/webhook/zhichi`（normalize→入队）。
  - `REST API`：工单查询、FAQ 管理、助手对话、Skill 管理（admin）、SSO 登录。
- **解耦**：webhook **只做"入队"不做处理**——KSM 存 noticeNum 入队后立即返回，重活交 worker，避免阻塞来源系统回调（KSM subscribeCallback 本身 60s 慢）。

### ② 业务层（Service / 编排）
- **职责**：业务编排、事务边界、**唯一读写 DB 的层**。Skill 不碰 DB（详规约定）。
- **核心**：`pipeline_runner`（harness）——`_run_step(name, fn, payload)` 包裹每步，统一写 `t_skill_log`，编排 S1~S13 与分支。
- **解耦**：业务层通过 `t_task_queue` 与 worker 解耦——API 进程入队，worker 进程消费跑流水线。两进程共用 Service 代码、各自连 DB。

### ③ AI 服务层（Skill Runner）
- **职责**：执行 9 个 SKILL（LLM 判断，从 `t_skill_md` 读热加载）+ py skills（确定性）。**Skill 纯函数，入 payload 出 result 信封 `{status, fields, evidence, model_used}`，绝不连 DB**。
- **解耦**：Skill 与业务层经 payload/result 信封解耦；LLM 供应商经"LLM 网关"解耦（Claude→DeepSeek 降级在网关内，业务无感）。

### ④ 第三方集成层（Integrations）
- **职责**：封装所有外部 HTTP（KSM/智齿/ask agent/硅基流动/DeepSeek/飞书/Linear/MinIO），统一超时/重试/降级（第8条矩阵）。
- **解耦**：每个外部系统一个 client 类，业务层只调 client 接口；超时/重试/token 缓存封装在 client 内，业务无感。

### ⑤ 数据层（Repository + PG + MinIO）
- **职责**：数据持久化。Repository 模式封装 SQL；PG 存业务+队列+向量；MinIO 存附件。
- **解耦**：Repository 隔离 SQL 与业务逻辑；向量检索（pgvector）封装为 repo 方法。

### ⑥ 定时任务层（Scheduler）
- **职责**：周期任务——SLA 扫描（每日北京 08:00，双 SLA）、观察期扫描（resolved+14天）、队列健康监控、节假日 API 同步。
- **解耦**：独立 scheduler 进程（或 APScheduler），通过入队或直接调 Service 执行；与 API/worker 进程分离，互不阻塞。

## 三、全局统一规范

### 3.1 统一返回体
```python
# 成功
{"code": 0, "message": "ok", "data": {...}}
# 失败
{"code": <业务错误码>, "message": "<可读信息>", "data": null}
```
- HTTP 状态码表达传输层（200/401/403/500），`code` 表达业务语义。
- 分页：`data: {items: [], total, page, page_size}`。

### 3.2 异常处理
- **异常分层**：`BizException(code, message)`（业务可预期）· `IntegrationException`（外部接口）· 未捕获异常（兜底 500）。
- **全局异常处理器**（FastAPI `exception_handler`）：统一捕获 → 转标准返回体 → 记日志（含 trace_id）。
- **Skill/流水线异常**：harness 兜底记 `t_skill_log.status=failed`，不中断 worker（详规约定）；任务重试 ≤3 → abandoned → 自动转人工（第8条）。
- **不向客户/外部泄露堆栈**：对外返回脱敏 message，详情进日志。

### 3.3 日志
- **结构化日志**（JSON），每条带 `trace_id`（贯穿一次 webhook→流水线全链路）+ `ticket_id`。
- **分级**：INFO（流程节点）· WARN（降级触发）· ERROR（异常）。
- **业务审计**：`t_skill_log`（每步结果+依据+模型+状态+耗时）· `t_operation_log`（SKILL.md 编辑前后值）。
- **敏感信息**：日志中 PII 按脱敏规则处理，不落明文密钥/密码。

### 3.4 参数校验
- **Pydantic 模型**校验所有入参（类型/必填/范围），FastAPI 自动 422。
- **业务校验**在 Service 层（如提单 AS-01：标题描述非空+≥15字+防重复）。
- **NL2SQL 护栏**（py 层强制）：SELECT-only + LIMIT 封顶 + PII 角色脱敏——独立校验器，不可热改。

### 3.5 事务
- **边界在 Service 层**：一个业务动作一个事务（async session）。
- **流水线分步提交**：每个 Skill step 完成即提交（落 result + skill_log），便于断点续跑、失败不回滚已完成步骤。
- **写操作 + skill_log 同事务**：保证审计与业务一致。
- **外部调用不在事务内**：HTTP 调用（回写/通知）放事务外，避免长事务持锁；回写状态用 `writeback_status` 跟踪 + 重试。

### 3.6 权限拦截
- **JWT 中间件**：解析 `Authorization: Bearer`，注入 `current_user(sub/name/role)`。
- **RBAC 三级**：admin（全表读写+Skill管理）· handler（业务读写）· visitor（只读+只读统计）。
- **依赖注入校验**：FastAPI `Depends(require_role("admin"))` 装饰需权限的路由。
- **数据级脱敏**：NL2SQL/统计结果按 role 对 PII 列脱敏（visitor 脱敏，handler/admin 明文）。
- **Skill 管理**：仅 admin 可编辑 9 个 SKILL.md（PUT 校验 frontmatter→存 t_skill_md→bump version→清缓存）。

## 四、关键架构决策与业务规则映射

| 业务规则 | 架构落点 |
|---|---|
| 流转判断 FR-05 全进 SKILL.md（无 py 护栏） | Skill Runner 从 t_skill_md 热加载，无独立护栏代码 |
| info-dedup 短路复用（第2条） | pipeline_runner 在 FAQ 前插入短路出口；pgvector 算相似度 |
| 双 SLA + 工作日（第7条） | scheduler 每日08:00 跑两套扫描；节假日 API 同步到 t_holiday |
| ask agent 四步异步（第8条） | 集成层 ask_agent_client；worker 轮询 60s×3 |
| 退回跑AI转人工（第1条） | pipeline_runner 末端按 is_returned 强制 pending_manual |
| 助手只读 NL2SQL（第9条） | assistant_service + py 护栏校验器 + assistant-nl2sql SKILL |
| 关键依赖全挂挂起（第8条） | worker 探活开关，暂停拉取队列 |

## 五、进程拓扑（Docker Compose）

| 服务 | 职责 | 副本 |
|---|---|---|
| `backend` | API + webhook（接入层，只入队） | 可多副本 |
| `worker` | 消费 t_task_queue 跑流水线 | 可多副本（SKIP LOCKED 抢锁） |
| `scheduler` | SLA/观察期/队列健康/节假日 定时 | **单副本**（防重复触发） |
| `postgres` | 业务+队列+向量（pgvector） | 单实例（内网） |
| `minio` | 附件/图片 | 单实例 + 对外 HTTPS 出口 |

> backend/worker 共用 Service+Skill 代码，配置区分角色启动。scheduler 单副本避免定时任务重复执行。
