# ticket-hub 数据库表结构设计（DDL）

> 编制：资深后端架构师 · 日期 2026-06-10
> 规范：三范式为主 + 高频字段合理冗余；PostgreSQL 16 + pgvector。
> 统一约定：主键 `id BIGSERIAL`；时间 `timestamptz`；表 `t_` 前缀 snake_case；枚举 `varchar`+注释（线上可加值）。

## 〇、统一审计字段（每表必带 5 字段）

| 字段 | 类型 | 默认 | 注释 |
|---|---|---|---|
| is_deleted | boolean | false | 逻辑删除 |
| created_at | timestamptz | now() | 创建时间 |
| updated_at | timestamptz | now() | 更新时间 |
| created_by | varchar(64) | NULL | 创建人（飞书uid/system） |
| updated_by | varchar(64) | NULL | 更新人 |

| customer_mobile | varchar(32) | | 联系手机（PII） |

| product_name_raw | varchar(128) | | 来源原始产品名 |
| module_raw | varchar(128) | | 来源原始模块名 |
| raw_json | jsonb | | 来源完整原始报文（明文留底，PII不进LLM，不建GIN仅溯源） |

| full_reply | text | | agent 全文（含【需要您提供】） |

| writeback_error | text | | 回写失败详情 |

## 表3 · t_ticket_tag（工单标签明细/历史表，1:N）

**用途**：记录工单**每次打标的明细与修正轨迹**（一单可多行）。主表只存当前生效的冗余标签值，本表存"标签怎么来的、置信度、LLM判还是人工改、改过几次"，供审计与打标质量分析。

| 字段 | 类型 | 约束 | 注释 |
|---|---|---|---|
| id | BIGSERIAL | PK | 主键 |
| ticket_id | bigint | NOT NULL | 关联 t_ticket_info（1:N） |
| product_tag | varchar(64) | | 产品线 |
| func_module | varchar(128) | | 功能模块 |
| dev_owner | varchar(64) | | 研发责任人（查表得，非LLM） |
| dev_owner_missing | boolean | NOT NULL DEFAULT false | 责任人未匹配 |
| confidence | numeric(4,3) | | LLM 置信度（0~1，可空） |
| tag_source | varchar(16) | NOT NULL DEFAULT 'llm' | 判定来源：llm/manual/fallback |
| model_used | varchar(32) | | 使用的模型（claude/deepseek） |
| is_current | boolean | NOT NULL DEFAULT true | 是否当前生效（最新一条 true） |
| revised_by | varchar(64) | | 人工修正人（manual 时） |
| evidence | text | | 判定依据/触发词命中 |
| *(+ 审计5字段)* | | | |

**索引**：
- `idx_ticket_current` (ticket_id, is_current) — 取当前生效标签
- `idx_ticket_created` (ticket_id, created_at) — 打标历史时间线

**说明**：重新打标时，旧行 `is_current=false`、插新行 `is_current=true`；主表冗余值同步刷新为最新。

## 表4 · t_ticket_merge（工单合并/关联记录表，1:N）

**用途**：记录工单的重复关联/合并事件。三类（merge_type 区分）：① `dedup_reuse` info 短路复用历史答复；② `dedup_marked` 判重复但目标无有效答复未复用（回退走流水线）；③ `hub_merge` bug 合并到研发单。留存相似度/阈值/LLM确认，供溯源与重复率分析。主表存当前值（reused_from_ticket_id/rd_hub_id），本表存明细历史。

| 字段 | 类型 | 约束 | 注释 |
|---|---|---|---|
| id | BIGSERIAL | PK | 主键 |
| ticket_id | bigint | NOT NULL | 当前工单 id（被判重复的） |
| merge_type | varchar(16) | NOT NULL | dedup_reuse / dedup_marked / hub_merge |
| target_ticket_id | bigint | | 复用目标工单 id（dedup_* 时） |
| target_hub_id | bigint | | 合并到的 hub id（hub_merge 时） |
| similarity | numeric(4,3) | | 相似度得分 0~1 |
| threshold | numeric(4,3) | | 当时阈值（0.85/0.7） |
| llm_confirmed | boolean | | LLM 二次确认结果 |
| evidence | text | | 判定依据（含"无有效答复未复用"等） |
| *(+ 审计5字段)* | | | |

**索引**：
- `idx_ticket` (ticket_id)
- `idx_target_ticket` (target_ticket_id) — 反查谁复用了我
- `idx_target_hub` (target_hub_id) — 查某 hub 合并了哪些单
- `idx_type` (merge_type) — 按类型统计重复率

## 表5 · t_ticket_hub（研发单主表，N info : 1 hub）

**用途**：研发要修的**一个问题（bug）**，独立于工单池——研发在 hub 页面只看待修问题，多个重复 bug 工单(info)经 hub-dedup 合并指向同一 hub。承载 SLA-2 研发计时 + Linear 同步。

| 字段 | 类型 | 约束 | 注释 |
|---|---|---|---|
| id | BIGSERIAL | PK | 主键 |
| hub_no | varchar(32) | NOT NULL UNIQUE | 研发单号 HUB+yyyyMMdd+seq |
| title | varchar(512) | NOT NULL | 标题 |
| problem_summary | text | | 问题归纳 |
| full_reply | text | | 完整答复（同步 Linear 用） |
| product_tag | varchar(64) | | 产品线 |
| func_module | varchar(128) | | 功能模块 |
| dev_owner | varchar(64) | | 研发责任人（SLA-2 通报对象） |
| embedding | vector(1024) | | hub 向量（hub-dedup 召回，bge-m3 维度） |
| linear_issue_id | varchar(64) | | Linear issue id |
| linear_url | varchar(512) | | Linear 链接 |
| linear_sync_status | varchar(16) | NOT NULL DEFAULT 'pending' | pending/synced/failed |
| linear_sync_retry | int | NOT NULL DEFAULT 0 | 同步重试次数 |
| **-- Linear 回调回写（处理人/状态/说明）** | | | |
| rd_handler | varchar(64) | | ✏️Linear 回调回写的研发处理人 |
| rd_status_note | text | | ✏️Linear 回调回写的处理说明 |
| rd_callback_at | timestamptz | | 最近一次 Linear 回调时间 |
| **-- SLA-2 研发（hub 创建起）** | | | |
| rd_sla_start_at | timestamptz | | =hub.created_at |
| rd_sla_due_at | timestamptz | | 到期（工作日+sla_rd_hours） |
| rd_sla_state | varchar(16) | NOT NULL DEFAULT 'normal' | normal/breached |
| rd_resolved_at | timestamptz | | 研发解决时间（Linear 回调 status=resolved 时填，SLA-2 停表） |
| status | varchar(20) | NOT NULL DEFAULT 'in_progress' | ✏️转研发后默认处理中：in_progress/resolved/closed（由 Linear 回调驱动） |
| *(+ 审计5字段)* | | | |

**索引**：
- `uk_hub_no` UNIQUE(hub_no)
- `idx_dev_owner_sla` (dev_owner, rd_sla_state, rd_sla_due_at) — SLA-2 扫描
- `idx_product` (product_tag) — hub-dedup 同产品线召回
- `idx_linear_status` (linear_sync_status) — 同步失败重试扫描
- `ivf_embedding` USING ivfflat(embedding vector_cosine_ops) — pgvector 相似度（hub-dedup）

**关系**：t_ticket_info.rd_hub_id → t_ticket_hub.id（N:1）。

**Linear 双向同步**：转研发→linear-sync(issueCreate)，hub 默认 `in_progress`；系统提供 **Linear 回调 webhook**，回写 rd_handler/status/rd_status_note→hub，并**镜像同步回 info**(rd_status/rd_handler/rd_status_note)；status=resolved 时填 rd_resolved_at 停 SLA-2。

## 表8 · t_sla_log（SLA 超时记录表，双 SLA）

| overdue_hours | numeric(8,2) | | 已超时时长（工作日小时） |
| notified | boolean | NOT NULL DEFAULT false | 是否成功通报 |

- `idx_type_date` (sla_type, notify_date) — 按类型统计

## 表9 · t_integration_log（第三方接口日志表，高写入）

| biz_success | boolean | | 业务是否成功（status==true/ret_code==000000） |
| error_msg | varchar(512) | | 失败信息 |
| duration_ms | int | | 耗时毫秒 |
| retry_seq | int | NOT NULL DEFAULT 0 | 第几次重试（0=首次） |
| *(+ 审计5字段)* | | | |

**索引**：
- `idx_trace` (trace_id) — 按链路查全过程
- `idx_ticket` (ticket_id) — 查某工单所有接口调用
- `idx_integration_created` (integration, created_at) — 接口健康/耗时分析

**留存**：✏️保留 120 天，超期定期清理（高写入表破例，区别于其他表长期保留）。异步写入（入队批量落库）。
