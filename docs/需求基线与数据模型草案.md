## 一、已锁定决策（产品方 2026-06-10 裁决）

| # | 决策项 | 结论 |
|---|--------|------|
| 1 | **拆分模型** | **不拆单**。一单一问题为处理单元；**"非一对一"（一单多问题）→ 不接管 → 退回**，不进入打标/答复流水线。 |
| 2 | **FAQ 检索源 + 命中处置** | **【v2修正】检索源 = 本地 `t_faq` + 硅基流动向量（>0.8 命中）**，不再用 Dify。命中 → **直接回写关单**（无需人工确认）。 |
| 3 | **关单回复内容** | 答复客户固定两段：**【问题根因】+【解决方案】**。转人工/转研发用完整内容（含【需要您提供】）。 |
| 4 | **ask agent 正式 cid** | **【v2修正】cid 由 `ask_init` 每会话动态生成**（非固定配置）；正式 appid/app_key/host 待联调。 |
| 5 | **FAQ 审核流程** | **先入知识库，后审核**：收录即写入**本地 t_faq**；DeepSeek 审核结果**只飞书通知审核人**，**不阻断入库**。 |
| 6 | **权限边界** | **admin / handler = 全部表读写**；**访客（visitor）= 只读**。三级即可。 |
| 7 | **SLA 计时口径** | ✏️**v2.1 工作日计时**（排除周末+法定节假日，接国务院节假日 API）；**双 SLA**（人工/研发）。工单 >对应时长未关闭 → 触发通报（见 R2）。 |
| 8 | **密钥/部署** | 密钥**一律走环境变量**；**服务器在内网**。 |
| 9 | **附件/图片存储** | 工单附件（全部类型）入站即转存 **MinIO**；FAQ 图片同存 MinIO。**【v2】对外访问出口 `https://fpy-jfsv.kingdee.com:8864/`**。 |
| 10 | **Skill 标准 + 页面编辑** | Skill 遵循 **Claude skill-creator 标准**；**页面可编辑 SKILL.md**（仅 LLM 判断类，9 个），保存热生效。 |

## 二、补充细则（v2 更新）

| ID | 议题 | 结论 |
|----|------|---------|
| **R1** | FAQ 审核不通过怎么办 | 审核 `rejected` → **仅飞书通知审核人，不自动撤回**；人工决定删除/修订。**【v2】检索时过滤 rejected**（只命中 approved+pending_review），杜绝错误 FAQ 自动回客户。 |
| **R2** | SLA 计时起点/通报 | ✏️**v2.1 双 SLA + 工作日**：SLA-1 人工(t_ticket_info)从 `created_at` 起 / SLA-2 研发(t_ticket_hub)**从 hub 创建时刻起**，时长各自可配(默认48)、**工作日计时**(排除周末+法定节假日，接国务院节假日 API)。**超时每天北京时间 08:00 一次**(去14:00)、按责任人**计数+单号列表**、SLA-1/SLA-2 分不同 webhook 群；bug 单 info 转研发后停 SLA-1 由 hub SLA-2 接力；观察期不跑 SLA。`dev_owner_missing` 归"未分配"桶。 |
| **R3** | 不接管时的来源系统动作 | **KSM 不接管** = 仅飞书通知 + `status=returned`，**对客户零回写**。**【v2】智齿不接管** = 飞书通知 + **固定话术回复 + `ticket_status=3` 关单**（有意不对称）。 |
| **R4** | 硅基流动的角色 | **【v2修正】硅基流动 = FAQ 检索主引擎**（query 向量 vs 本地 t_faq，>0.8 命中）+ info-dedup(0.8)/hub-dedup(0.7) 召回。本地 `t_faq` 为**检索主库**。 |
| **R5** | 两段回复的取数 | `final_reply` = ask 模板【问题原因】→【问题根因】、【解决建议】→【解决方案】；FAQ 命中直接用 t_faq 正文。`full_reply` = ask 全文（含【需要您提供】）。**【v2】D 答复回写前过 reply-humanize 去AI化 → desensitize 脱敏**。 |
| **R6** | 通报渠道 | **飞书群机器人**；**【v2】统一一个可配置 webhook（`FEISHU_BOT_WEBHOOKS` 支持多个）+ 按场景模板匹配**，地址走配置不硬编码。 |
| **R7** | 重复推送 / 退回（v2.1 修正） | **推送编号 `source_id` 已存在 = 退回**：更新原单(不变号) + `return_count+1` + 退回标签 + 飞书通知人工。✏️**v2.1：退回单照常做流转判断 + 照常跑 AI 流水线**(生成 final_reply 作人工参考)，**末端一律转人工**(`pending_manual`)，不自动回写客户。 |
| **R8** | 关闭生命周期（v2 新增） | 答复关单 = 回写来源系统到**处理完成**(KSM 4/智齿 3) + 记 `resolved_at`；**14 天(可配)无退回 → 本地置 `closed`，不再回写来源系统**。KSM/智齿都有观察期，**assistant 无观察期**。 |
| **R9** | assistant 来源（v2 新增） | 新增 `source=assistant`，进完整流水线、计 SLA；**答复/话术经飞书私信推送提单人**（身份靠 SSO `feishu_uid`）。 |
| **R10** | 智能助手（v2 新增） | **本期只读 NL2SQL 统计**：SELECT-only + 全表 + LIMIT 封顶 + PII 列按角色脱敏；写操作降为后续规划。 |

## 三、需求基线流水线（v2）

```
[入站] 智齿 webhook(完整工单) / KSM webhook(noticeNum→subscribeCallback 拉全量) / assistant 提单
   → normalize(HTML清洗/时间戳/附件→MinIO) → 幂等(source,source_id)
        │  ★ 编号已存在 → 【退回】更新原单+退回标签+return_count+1+pending_manual+飞书通知人工 → ✏️v2.1 照常跑AI(作人工参考)+末端转人工
        │  ★ 新编号 → 入库 t_ticket_info → 入队 run_pipeline
        │  ★ SLA 时钟启动：sla_start_at=created_at, sla_due_at=+48h
        ▼
步骤1 ticket-routable（流转判断，不拆单）[SKILL+关键词护栏]
   ├ 非一对一/不可流转 ──→ 【不接管】
   │       ├ KSM：飞书通知 + status=returned（对客户零回写）
   │       └ 智齿：飞书通知 + 固定话术回复 + ticket_status=3 关单
   └ 可流转 ──→ KSM: lockKsmOrder 接管 → 继续
        ▼
步骤2 ticket-tagging[SKILL]：产品线 + 功能模块（dev_owner harness 查表，无映射→默认接口人）
        ▼
步骤③ info-dedup[SKILL：✏️短路出口 embedding召回+LLM确认, 0.85]
   └ 命中 → 复用历史最近有效答复 → writeback关单 → resolved_at → 结束（不走后续）
        ▼
步骤3 答复处理：
  3.1 desensitize(query) → faq-retrieve[py：硅基流动向量检索本地 t_faq, >0.8, 过滤rejected]
        └ 命中 → final_reply=FAQ正文 → writeback 关单 → 记 resolved_at → 结束
  3.2 desensitize(question) → agent-answer[py：open-api-channel 四步异步, 每工单一会话]
        │ transfer_result=TRANSFER → 直接分支C
        ▼
  3.3 answer-router[SKILL：NO_ACTION 时 LLM 语义分类；无法识别→C]:
        ┌ A bug      → hub-dedup[SKILL 0.7] → 重复:关联 rd_hub_id / 新建 hub + linear-sync(CNPRD)
        │              → 不回写客户, status=pending_rd（SLA 不豁免，照常通报）
        ├ B 需求     → status=pending_manual + 飞书群通知（人工转研发）
        ├ C 补充资料 → status=pending_manual + 飞书通知 / KSM supplyKsmOrder / 智齿统一回复
        └ D 正常回答 → reply-humanize[SKILL 去AI化] → desensitize(脱敏)
                       → final_reply【问题根因+解决方案】writeback 关单 → 记 resolved_at
                       → faq-record[SKILL 用humanize后版本归纳] → 写本地 t_faq(pending_review)
                       → faq-review[SKILL DeepSeek] → 仅飞书通知审核人（不阻断）
        ▼
[SLA 扫描器] 每日北京08:00(工作日)：SLA-1人工(ai_dev_owner)+SLA-2研发(hub dev_owner) 分别扫 → 计数+单号列表 → 分群通报
[观察期扫描器] resolved_at+14天 且未退回 → 本地 status=closed（不回写来源系统）
[队列健康监控] 积压告警 / 依赖全挂挂起 / abandoned 告警+手动重入队
```

assistant 来源：无外部系统回写，各节点"回写/话术"统一改为**飞书私信提单人**；D 答复直接 done（无观察期）。

## 四、数据模型草案（共享库"线上只加不删"，全部 `t_*` 前缀）

### t_ticket_info（业务主单 = 处理单元）
```
id, ticket_no(FPY+yyyyMMdd+seq, 用 PG sequence 防重号)
source(ksm|zhichi|assistant)  ★v2加 assistant
source_id, source_bill_no, raw_json(明文留底, PII不进LLM)
title, description, product_name_raw, module_raw, has_attachment, attachment_urls(jsonb)
customer_company, customer_contact, customer_mobile, customer_email
assistant_submitter_uid  ★v2: assistant来源的飞书提单人(私信回复用)
-- 流转
pipeline_status(routing|tagging|answering|done|returned|pending_manual|pending_rd)  ★v2加 pending_rd
routable(bool), route_action(可流转|不接管-多问题|不接管-不可流转), route_reason
-- 退回  ★v2新增
return_count(int default 0), is_returned(bool)  -- 编号已存在=退回, ✏️v2.1 照常跑AI作参考+末端转人工
-- 打标
ai_product_tag, ai_func_module, ai_dev_owner, dev_owner_missing(bool)  -- 无映射→默认接口人
-- 去重(✏️v2.1 短路出口)
is_duplicate(bool), dup_ticket_ids(jsonb), reused_from_ticket_id  -- 命中复用历史答复关单
-- 答复
faq_hit_id, faq_score, answer_branch(A|B|C|D), transfer_result(NO_ACTION|TRANSFER)  ★v2加
ai_reply(agent原文), full_reply, humanized_reply(去AI化后)  ★v2加 humanized_reply
final_reply(问题根因+解决方案, =humanize+脱敏后)
rd_hub_id(fk t_ticket_hub)
-- 回写
writeback_status, writeback_error, writeback_retry(int default 0)
ksm_takeover_account, ksm_takeover_name, ksm_takeover_number
-- SLA + 关闭生命周期
sla_start_at, sla_due_at, sla_state(normal|breached)
sla_notified_marks(jsonb)  ★v2: 通报时点列表(按时点去重, 替代 sla_notified_levels)
resolved_at  ★v2: 回写处理完成时间(观察期计时起点)
closed_at  -- 本地置closed时间
status(pending|returned|pending_manual|pending_rd|done|closed)  ★v2加 pending_rd
created_at, updated_at
```

## 五、对外接口契约要点（v2）

| ~~FAQ检索/写入~~ | ~~Dify~~ | ✏️v2 **整个移除** |
