## 第 1 条 · 工单流转判断与无效工单过滤规则 ✅

> 对应 Skill：`ticket-routable`（SKILL.md，LLM 判定，**全部规则含关键词/模块清单页面可热改**）+ 幂等/退回（py）。
> 触发时机：入站 normalize 入库后、打标前的**第一道闸门**。
> ⚠️ v2.1 变更：FR-05 取消 py 硬护栏，整体并入 SKILL.md（全页面可控，无写死兜底）；FR-01 退回改为"跑完 AI 后默认转人工"。

### 一、判定流程（退回单与新单分流）

```
入站工单
  ├ FR-01 编号已存在(source_id) → 【退回单】
  │     → 照常做流转判断(FR-02~05) + 照常跑 AI 流水线(打标/检索/agent/分类生成 final_reply)
  │     → AI 结果仅作人工参考，末端【一律转人工 pending_manual】，不自动回写客户
  │     → return_count+1 + 退回标签 + 飞书通知人工
  └ FR-01 新编号 → 【新单】走下方流转判断(命中即停)
```

| 顺序 | 规则ID | 检查 | 命中结果 |
|---|---|---|---|
| P1 | **FR-02** | 无效：描述 <15 字 **且** 无附件 **且** LLM 判语义不明 | 不接管-不可流转 |
| P2 | **FR-03** | 非一对一：一单多问题 | 不接管-多问题 |
| P3 | **FR-04** | 非发票云模块：命中黑名单 **且** 无发票云关键词 | 不接管-不可流转 |
| P4 | **FR-05** | 完全无发票云关键词 | 不接管-不可流转 |
| P5 | **FR-06** | 都不命中 | 可流转，进打标 |

### 二、逐条规则定义

**FR-01 幂等/退回**（py）
- 条件：`(source, source_id)` 在 `t_ticket_info` 已存在。
- 动作：✏️**判退回 → 照常做流转判断 + 照常跑 AI 流水线**（生成 final_reply 作人工参考）→ **末端一律转人工**（`status=pending_manual`，不自动回写客户）→ `return_count+1` + 退回标签 + 飞书通知。
- 不区分原单状态；若退回单流转判断为不接管，则按不接管处理（不再跑后续）。

**FR-02 无效/垃圾过滤**（LLM + 规则）
- 条件：`description` 去空白 **< 15 字** **且** `has_attachment=false` **且** LLM 判定语义不明/不足以处理。
- ✏️ 三条件**同时满足**才过滤（字数短但语义清楚的不杀）。
- 判定：`不接管-不可流转`。

**FR-03 非一对一（多问题）**（LLM）
- 条件（任一）：① ≥2 个显式问题编号（"1." "2."/"①②"/"问题一/二"）；② 分隔语（"另外/还有/第二个问题"）后**接 ≥10 字新问题**（✏️阈值 6→10）。
- 判定：`不接管-多问题`。
- 依据话术：与智齿不接管话术"多个问题请拆分后提"一致。

**FR-04 非发票云模块**（LLM + 清单）
- 条件：`module_raw` 命中模块黑名单 **且** 全文无发票云关键词。
- 模块黑名单（🟡 待业务确认，✏️**写在 SKILL.md 内，页面可改**）：应收/应付/总账/生产/工资/HR/供应链/采购订单。
- 判定：`不接管-不可流转`。

**FR-05 无发票云关键词**（LLM，✏️取消 py 硬护栏）
- 条件：全文（标题+描述）完全不含发票云关键词。
- 关键词清单（🟡 待业务确认，✏️**写在 SKILL.md 内，页面可改**）：发票/开票/收票/影像/税控/金税/乐企/全电/数电/红冲/抵扣/查验。
- 判定：`不接管-不可流转`。
- ✏️ **v2.1：判定逻辑 + 清单全部在 SKILL.md，页面可热改，无代码兜底闸**。

**FR-06 可流转**
- 条件：FR-02~FR-05 均未命中。
- 动作：`routable = true`，进入打标（步骤2）；**KSM 来源**先调 `lockKsmOrder` 接管。

### 三、不接管后的来源系统动作（按来源不对称）

| 来源 | 不接管动作 |
|---|---|
| **KSM** | 飞书通知 + `status = returned`，**对客户零回写**（不调任何 KSM 接口） |
| **智齿** | 飞书通知 + **固定话术回复** + `ticket_status = 3` 关单 |
| **assistant** | 飞书私信提单人告知话术 |

- 智齿固定话术（走配置）："您好，当前工单存在非发票云问题，请通过 KSM 提单提交对应模块处理人；或者工单中有多个问题，请拆分后提。"

### 四、降级与异常

| 场景 | 处置 |
|---|---|
| Claude 失败 | 降级 DeepSeek |
| ✏️ **Claude + DeepSeek 都失败** | **任务挂起等待**（入 `t_task_queue`，依赖恢复后重判；与 Q19 一致）——不猜测、不误放 |

> ✏️ v2.1：原"关键词规则兜底"随 py 护栏一并取消；依赖全挂改为挂起而非降级硬判。

### 五、输出字段

`routable(bool)` · `route_action(可流转|不接管-多问题|不接管-不可流转)` · `route_reason(判定依据文本)` · `is_returned(bool)` · `return_count`

## 第 2 条 · Skill 能力规则（问题过滤 / 重复合并 / 数据留存）✅

> 对应 Skill：`info-dedup`（SKILL.md，✏️v2.1 **短路出口**，非旁路）· `hub-dedup`（SKILL.md，研发单层合并）。
> ✏️v2.1：info 层不物理合并，但**命中重复即短路**——复用历史答复直接回写关单，不走后续流水线。

### 一、SK-01 问题过滤（去重前置）

- 不接管单已出局，不进 info-dedup；退回单照常转人工（去重仅作参考）。
- 候选范围：**同产品线**（`ai_product_tag` 相同）+ **近 120 天**（可配）。

### 二、SK-02 info-dedup（✏️工单层短路出口，不物理合并）

| 项 | 规则 |
|---|---|
| **位置** | ✏️**打标后、FAQ 检索之前**（短路闸，优先级高于 FAQ；历史完整答复优于通用 FAQ） |
| 召回 | 硅基流动 embedding 取同产品线近 120 天 **top8** 候选 |
| 判定 | ✏️**阈值 0.85**（可配，从 0.8 调高，因复用完整答复从严）召回 + LLM 二次语义确认"是否真重复" |
| **复用对象** | 重复单中**有有效 final_reply 且已成功关单(done/观察期)的、时间最近的一单** |
| **命中动作** | ✏️**短路**：直接复用该历史单 final_reply → 回写当前工单关单 → 记 resolved_at → **结束**（不调 agent、不分类、不收录 FAQ） |
| **回退** | ✏️若重复单中**无任何有效答复**（都挂着/转人工）→ **不短路**，当前工单照常走流水线 |
| 物理合并 | ✏️**不合并**：两张工单各自存在，仅当前单 `dup_ticket_ids` 记重复来源 + 短路复用 |
| 输出 | `is_duplicate(bool)` / `dup_ticket_ids(jsonb)` / `reused_from_ticket_id` |
| 异常 | 硅基流动挂 → 降级"非重复"，照常走流水线 |

### 三、SK-03 hub-dedup（研发单层合并，分支 A bug）

| 项 | 规则 |
|---|---|
| 触发 | 分支 A（bug）转研发前 |
| 召回 | 硅基流动取历史 hub **top5** 候选 |
| 判定 | **阈值 0.7**（可配）召回 + LLM 二次语义确认"是否同一研发问题" |
| **合并逻辑** | 确认重复 → `rd_hub_id` **关联已有 hub**（N info : 1 hub），**不新建 Linear**；不重复 → **新建 hub** + linear-sync(CNPRD) |
| 输出 | `is_dup(bool)` / `dup_hub_id` / `score` |
| 异常 | 硅基流动/LLM 挂 → 降级"非重复"（新建），避免漏单 |

### 四、SK-04 数据留存规则

| 数据 | 留存策略 |
|---|---|
| `t_ticket_info` / `t_ticket_hub` / `t_faq` | **线上只加不删**，软状态流转，不物理删除 |
| `dup_ticket_ids` / `reused_from_ticket_id` | 工单上长期保留，供溯源 |
| `raw_json`（含 PII 明文） | ✏️**长期留底**（内网安全边界） |
| `t_skill_log` | 每步留痕，长期保留供审计 |
| `t_attachment`（MinIO） | ✏️**长期留存** |

> ✏️v2.1：留存策略统一**长期保留**（不设自动清理/归档），依赖内网 + DB 访问控制兜底合规。

## 第 4 条 · 知识库规则（FAQ 检索 / 重试，✏️无 Dify）✅

> 对应：`faq-retrieve`（py，硅基流动向量检索本地 t_faq）+ 收录侧 embedding。
> ✏️ **Dify 已整体移除**，本条无"双库场景区分"；FAQ 全本地化（硅基流动检索 + DeepSeek 审核）。

### 一、KB-01 FAQ 检索（硅基流动向量）

| 项 | 规则 |
|---|---|
| 流程 | 脱敏 query → 硅基流动 embedding → 与本地 `t_faq.embedding` 余弦相似度 → 取最高分 |
| 命中阈值 | **> 0.8**（可配 `faq_hit_threshold`）→ 命中即用 FAQ 正文回写关单、记 resolved_at、结束 |
| 状态过滤 | 只比 `approved` + `pending_review`，**排除 `rejected`** |
| 候选范围 | ✏️**限同产品线**（`ai_product_tag` 相同） |
| embedding 模型 | ✏️**bge-m3**（中文语义+长文本，硅基流动提供）；模型名走配置 `embedding_model`，可切换 |

### 二、KB-02 检索重试 / 降级

| 场景 | 处置 |
|---|---|
| 硅基流动超时/失败 | ✏️**重试 3 次（指数退避，与全局统一）** → 仍失败 → **视为未命中**，进 ask agent（不阻断主流程） |

### 三、KB-03 检索顺序（与去重协同，第2条已定）

`info-dedup（短路复用历史答复 0.85）→ FAQ 检索（0.8）→ ask agent`。
- 阈值梯度合理：info（同工单重复）0.85 从严 > FAQ（通用问答库）0.8 较宽松。

### 四、KB-04 FAQ 写入侧 embedding

- 分支 D 收录时，用 **humanize+脱敏后**的内容算 embedding，落 `t_faq.embedding`，供 KB-01 检索。

### 五、KB-05 收录前去重

- faq-record 前用硅基流动 **0.85** 与**同产品线** t_faq 比对，≥0.85 → 不收录（已有类似，`faq_status=no_faq` 终止）。

### 六、阈值汇总（全部可配）

| 用途 | 阈值 | 范围 |
|---|---|---|
| info-dedup 短路复用 | 0.85 | 同产品线近120天 top8 |
| FAQ 检索命中 | 0.8 | 同产品线，过滤 rejected |
| FAQ 收录去重 | 0.85 | 同产品线 |
| hub-dedup 合并 | 0.7 | 历史 hub top5 |

## 第 5 条 · Agent 结果分类规则（A/B/C/D）🟡 待 agent 示例细化

> 对应 Skill：`answer-router`（SKILL.md，语义判定）。
> ⚠️ 本条基于当前理解，待用户提供 agent 真实答复示例后精修判定标准。

### 一、分类逻辑（✏️两层：先看 agent 类别语义，再从"正确回答"细分 C/D）

```
agent 返回(transfer_result + answer)
  ├ transfer_result=TRANSFER ──────────────→ C（agent 主动转人工，最高优先）
  └ NO_ACTION → 看 answer 类别语义：
        ├ 含明显 bug 语义（如"bug问题"）──→ A（Bug，转研发）
        ├ 需求 语义 ───────────────────→ B（需求，转人工→研发）
        └ 正确回答 语义 → 二次细分：
              ├ 答复中需要客户提供更多信息才能解决 → C（补充资料）
              └ 否则（已给可操作解决方案）─────────→ D（正常问答）
```

✏️ **关键**：C 不是第一层独立分类，而是从"正确回答"里二次细分——agent 不会主动标"补充资料"，由我方判断"正确回答"里是否在要客户补信息：要补→C，不要→D（对应模板【需要您提供】段）。

### 二、四类定义

| 分支 | 类别 | 判定依据 | 动作（第1轮已定） |
|---|---|---|---|
| **A** | Bug | agent 答复含明显 bug 语义（如"bug问题"） | 不回写客户，hub-dedup→linear，`pending_rd` |
| **B** | 需求 | agent 答复为需求语义 | 转人工→研发，`pending_manual` |
| **C** | 补充资料 | "正确回答"中**需客户提供更多信息**才能解决 | 转人工/补料话术，`pending_manual` |
| **D** | 正常问答 | "正确回答"中**已给可操作解决方案** | reply-humanize→脱敏→回写关单→收录FAQ |

### 三、兜底

- `transfer_result=TRANSFER` → C（最高优先）。
- 无法识别/LLM 失败 → **兜底 C**（转人工）。

### 四、待确认 🟡（等 agent 示例）

1. **A bug 语义识别**：靠 agent 返回的明显标识词，还是 LLM 语义判断？（等示例看 agent 实际标记形式）
2. **C/D 冲突边界**：答复既给方案、又附"如方便请提供截图"这种**非必须补充**——判 C 还是 D？建议："缺信息就无法解决"才判 C；"补了更好但已能解决"判 D。等示例确认。
3. 用户将提供**多个 agent 答复示例**，据真实样本细化以上判定。

## 第 6 条 · FAQ 收录 & 审核规则 ✅

> 对应 Skill：`faq-record`（归纳）+ `faq-review`（DeepSeek 审核）+ `desensitize`（脱敏）。

### 一、FQ-01 收录触发与取数

| 项 | 规则 |
|---|---|
| 触发 | 仅**分支 D 正常问答成功答复后**收录 |
| 取数 | 用 **humanize + 脱敏后**的 `description + final_reply` 归纳 `{title ≤20字, content ≤300字}` |
| 收录前去重 | 硅基流动 0.85 与**同产品线** t_faq 比对，≥0.85 → 不收录（`faq_status=no_faq`） |

### 二、FQ-02 脱敏字段（desensitize，✏️扩充 + 两类脱敏方式）

调用点：① 调 ask agent 前(question) ② 答客户/回写前(answer) ③ 收录 FAQ 前。

**① 完全替换类**（敏感值整体抹为占位符如 `[已脱敏]`）
- 替换：公司名 `customer_company`、联系人 `customer_contact`。
- 单号/编号：单号(FPY/HUB/ZC/R)、**外部供应商订单号、合同号、账号、authCode**（如 `4DLW2D124`）。
- 邮箱、座机、税号。
- ✏️**凭证类（按字段名识别，抹掉值）**：`clientSecret` / `entryKey` / `appSecret` / `privateKey` / `password` / `token` / `secret`、代码/ID/密钥串。

**② 部分保留打码类**（保留首尾，中间打码，便于辨认）
- ✏️**完整手机号** → 前 3 后 4，中间 `****`（`138****5678`）。
- ✏️**完整身份证号** → 前 6 后 4。

> 手机号统一以"前3后4打码"为准（覆盖原"整体抹除"）；出站答复/FAQ收录/agent入参三处一致。

### 三、FQ-03 收录入库 + embedding

- ✏️**先写本地 t_faq**（`review_status=pending_review` + 算 embedding）→ **再**触发审核。
- **先入库后审核、审核不阻断**：pending_review 可被检索（已成稿质量有底），仅 rejected 排除。

### 四、FQ-04 DeepSeek 审核维度（四维）

| 维度 | 检查 |
|---|---|
| ① 敏感信息残留 | 脱敏是否干净（有无漏网 PII/密码/账号） |
| ② 事实性 | 答复是否正确、无误导 |
| ③ 内部细节泄露 | 有无内部系统/代码/人员/架构信息 |
| ④ 表达质量 | 是否清晰、适合入库 |

### 五、FQ-05 通过 / 驳回标准

| 结果 | 条件 | 动作 |
|---|---|---|
| `approved` | 四维**全过** | `review_status=approved` |
| `rejected` | **任一维不过**（不分轻重） | ✏️**方案 A**：`review_status=rejected` → **检索自动排除（软下线，不再回客户）** + 飞书通知审核人 → **不自动删数据**，人工决定删/改/重审 |

### 六、FQ-06 飞书通知规则

- **仅 rejected 通知**审核人（approved 不通知，防刷屏）。
- 内容：FAQ 标题 / 原工单号 / 驳回维度 + reason。
- ✏️走统一可配 `FEISHU_BOT_WEBHOOKS`（可多个）+ 模板；@审核人名单 🔗 占位待补（L6）。

## 第 7 条 · SLA 规则（✏️双 SLA + 工作日计时）✅

> ✏️v2.1 重大修正（覆盖 Q1）：① **双 SLA**（人工 + 研发）；② **工作日计时**（排除周末+法定节假日）；③ 通报**仅 08:00 一次**、**计数+单号列表**。

### 一、SLA-00 双 SLA 模型 🆕

| | **SLA-1 人工处理** | **SLA-2 研发处理** |

| 对象 | `t_ticket_info`（工单） | `t_ticket_hub`（研发单） |
| 时长 | `sla_manual_hours`（默认 48，可配） | `sla_rd_hours`（默认 48，可配） |

| 通报群 | 人工 SLA 群 webhook | 研发 SLA 群 webhook（不同群） |

### 二、SLA-01 计时口径

| 项 | 规则 |
|---|---|

| 到期 | `sla_due_at = created_at + sla_manual_hours`（工作日累加） | `rd_sla_due_at = rd_sla_start_at + sla_rd_hours`（工作日累加） |
| 停表 | 到达终态 `done/closed/returned`，或 `resolved_at`（处理完成即停） |

### 三、SLA-02 暂停场景

- ✏️**全程不暂停**：转人工(B)/补料(C)/转研发(A) 期间 SLA 继续跑（含等客户补料）。

### 四、SLA-03 bug 单的 SLA 接力（✏️双 SLA 衔接）

| 形式 | ✏️按责任人分组，**计数 + 单号列表**（"你名下 N 个超时：FPY-xxx, FPY-yyy…"） |
| SLA-1 通报 | 人工处理人 / `ai_dev_owner`（`dev_owner_missing`→"未分配"桶）→ 人工 SLA 群 |
| SLA-2 通报 | hub `dev_owner` → 研发 SLA 群 |
| 观察期 | ✏️**观察期（resolved 后 14 天）不跑 SLA**（resolved 即停表） |

### 六、SLA-05 标记逻辑

- `sla_state`：normal / breached（筛选用，不防重发）。
- `sla_notified_marks(jsonb)`：记已通报时点（按"日期+08:00"去重，防扫描器重入）。

## 第 8 条 · 第三方接口兜底矩阵 ✅

> 统一原则：超时分级（HTTP 30s / LLM 60s / agent 见下）；重试 ≤3 次指数退避；降级链清晰；abandoned 自动转人工兜底。

### 一、IF-01 各接口兜底矩阵

| 接口 | 超时 | 重试 | 降级 / 异常处置 |
|---|---|---|---|
| **KSM** subscribeCallback | 60s | ≤3 退避 | 仍失败 → abandoned（→ 自动转人工，见 IF-03） |
| **KSM** lock/handle/supply | 30s | ≤3 | 401/未授权→刷 token 重试1次；回写失败→`writeback_error`+重试≤3 |
| **智齿** get_token/reply/upload | 30s | token 失效刷新重试1次 + ≤3 | 回写失败→重试≤3；`ret_code≠000000`→记错误 |
| **硅基流动** embedding | 30s | **≤3 退避** | 检索→视未命中走 agent；去重→降级"非重复" |
| **ask agent** 四步异步 | ✏️轮询**每 60s 一次、最多 3 次**(总~180s) | token 失效刷新重试 | 轮询 ERROR/超时(3次未DONE)/`100002`/空 → **转分支 C**；`TRANSFER`→C |
| **Claude** | 60s | — | 降级 DeepSeek；都失败→routable 挂起 / 其他 Skill 记 failed 不阻断 |
| **DeepSeek** | 60s | — | Claude 降级目标；自身失败→记 failed 不阻断 |
| **飞书** webhook | 30s | ≤3 | 通知失败重试；同工单+同场景短时去抖防刷屏 |
| **Linear** issueCreate | 30s | ≤3 | hub 保留+`linear_sync_status=failed`重试；超限→飞书通知人工 |

> 超时/重试次数均走配置，可按接口微调。

### 二、IF-02 全局背压（Q19）

- **队列积压**（待处理 >N 或 最老等待 >M 分钟，走配置）→ 系统级飞书告警（独立群，与 SLA 通报区分）。
- **关键依赖全挂**（ask agent + DeepSeek 都不可用）→ 任务**挂起不消费、暂停拉取**，探活恢复后自动继续，不耗光重试。

### 三、IF-03 abandoned 处置（✏️自动转人工兜底）

- 任务重试耗尽（≤3）→ `abandoned`：
  - ✏️**自动置 `status=pending_manual` + 飞书通知人工**（兜底，防工单卡死无人管）。
  - **后台手动"重新入队"**口子。

### 四、降级链总览

```
LLM 判断类:  Claude → DeepSeek → (routable)挂起 / (其他)failed不阻断
FAQ 检索:    硅基流动 → 重试3次 → 视未命中 → ask agent
ask agent:   失败/超时/TRANSFER → 分支 C
回写/Linear: 重试≤3 → 失败通知人工
依赖全挂:    挂起不消费 → 探活恢复
abandoned:   自动 pending_manual + 通知 + 手动重入队
```
