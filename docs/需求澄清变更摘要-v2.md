## 一、决策清单（Q1~Q20 逐条）

### Q1 · SLA 计时与通报 ✏️🆕

| 项 | 决策 |
|---|---|
| 计时口径 | 7×24 自然计时，入站起表，`sla_due_at = sla_start_at + 48h` |
| 通报触发 | 超 48h 后**每天 08:00 / 14:00 两次**，无限直到工单关闭 |
| 通报形式 | ✏️ **按 `ai_dev_owner` 分组只报数量**（"你名下有 N 个超时未关闭"），非逐单刷屏 |
| 无责任人兜底 | `dev_owner_missing=true` 的归 **"未分配"桶**，单独发给固定接收人 |

> ✏️ 修正原基线 R2："超时仅通报一次、靠 `sla_state=breached` 防重发" → 改为**定时重复通报**，`sla_state` 退化为筛选条件，通报去重改为按时点。

### Q2 · 智齿侧状态机（相对 KSM 主流程的差异）🆕

主流程（步骤1→2→3.1/3.2/3.3 四分支）是 **KSM 标准流程**；智齿为变体，差异如下：

| 节点 | KSM（主流程） | 智齿（变体） |
|---|---|---|
| 不接管 | 不回写，流程结束 | 🆕 **飞书通知 + 固定话术回复客户 + `ticket_status=3` 关单** |
| 需求转人工(B) | `pending_manual`，不回写 | 同（不回写） |
| 补充资料(C) | `supplyKsmOrder` 平台补料 | 🆕 **`save_ticket_reply` 发"需要您提供…"**（统一回复） |
| 关单(B3/D) | `handleKsmOrder` 推进节点 | `save_ticket_reply` `ticket_status=3` |

- **智齿不接管固定话术**（走配置）：
  > "您好，当前工单存在非发票云问题，请通过 KSM 提单提交对应模块处理人；或者工单中有多个问题，请拆分后提。"
- ✏️ **KSM 不接管 = 对客户零回写**（有意与智齿不对称，不补话术）。

### Q3 · 分支 A（Bug）处理 🆕✏️

- bug 去重 → 同步 Linear → **不回写客户**，状态 🆕 **`pending_rd`（新增）**，等研发处理后人工关单。
- SLA **不豁免** bug 单：照常进 `ai_dev_owner` 每日两次超时计数通报（即催研发闭环手段）。

> 🆕 新增状态 `pending_rd`（等研发），与 `pending_manual`（等人工补料/需求）区分。

### Q4 · 知识库架构 ✏️（重大变更）

- ✏️ **移除 Dify 整个集成**（FAQ 库 + 产品/配置文档库**双库都去掉**）。
- ✏️ **"产品及配置文档库"整体删除**，不参与流水线、不在助手域对接；ask agent 答复能力以其内部为准，我方不喂文档库上下文。
- FAQ 全链路本地化：
  - **检索**：硅基流动 embedding，query 向量 vs 本地 `t_faq` 向量，**相似度 > 0.8 命中** → 直接回复关单。
  - **存储**：本地 `t_faq`（含 embedding，重新成为检索主库）。
  - **审核**：DeepSeek（不变）。

> ✏️ 修正原基线 R4/裁决2（FAQ 走 Dify hit-testing）；`t_faq` 从"审核镜像" → "检索主库"。

### Q5 · 分支路由方式 ✏️

- ✏️ 不靠 agent 返回首行关键词字面匹配，改为**我方语义判定 Skill**（Claude→DeepSeek，可热改）读 agent **全文**归类。
- 路由规则：bug→A / 需求→B / **【需要您提供】为核心、问题无法定位→C** / 正常回答→D；不确定/出错 **兜底 C**。
- （Q20 强化）先看 agent 的 `transfer_result`：`TRANSFER`→直接 C；`NO_ACTION`→再语义分类。

### Q6 · FAQ 检索状态过滤 🆕

- 检索本地 `t_faq` 时**只命中 `approved` + `pending_review`，排除 `rejected`**。审核拒绝的 FAQ 永不被自动回复客户（消解原盲点 D-1）。

### Q7 · 智能助手数据操作（本期范围见 Q17-19，此条降级）

- 原定：受限写（白名单表/字段、禁删除、不碰 users/权限/审计表）+ **所有写操作二次确认** + 自动提议（自动发起、人工确认）。
- ✏️ **Q17-19 收窄：本期助手只读、不做任何写操作**，本条整套写设计**降级为后续规划**。

### Q8 · assistant 来源工单 🆕

- 🆕 新增来源 `source=assistant`，进完整流水线、**计 SLA**。
- 无外部来源系统可回写，**所有答复/话术通过飞书私信推送给提单人**（身份靠 SSO 的 `feishu_uid`）：
  - FAQ 命中 / D 答复 → 私信发答复 + `status=done`（无观察期，见 Q13）。
  - 不接管 → 私信告知话术。
  - C 补料 → 私信发"需要您提供…"。
  - A bug / B 需求 → 私信告知"已转研发/人工" + 挂起（A→`pending_rd`，B→`pending_manual`）。

### Q9 · 飞书 SSO ✏️

- 照搬 `jfservice-auto/app/api/auth.py`：OAuth 授权码模式 —— `/feishu/login` 出授权 URL → `/feishu/callback`：`app_access_token` → `user_access_token`(oidc) → `user_info` → 按 `feishu_uid`（兜底姓名）同步 `t_users` → 签发 JWT。
- 全新用户默认 `visitor`；禁用账号拒登。凭证在参考项目 `.env` 已具备（`FEISHU_APP_ID/SECRET/REDIRECT_URI`）。

### Q10 · 流转判断规则 ✅

- 按推导 4 类"不接管"实现：①非一对一(多问题) ②描述<10字且无附件 ③非发票云模块且无发票云关键词 ④完全无发票云关键词；否则可流转。
- LLM 判定（可热改 SKILL.md）+ **关键词护栏硬兜底**（无发票云关键词强制不接管）。关键词/模块清单经业务确认权威。

### Q11 · 去重阈值 ✅

- info-dedup 0.8 / hub-dedup 0.7 / FAQ 检索 0.8，**全部走配置文件**，先用经验值上线，攒数据后校准。

### Q12 · 重复/更新推送 → 统一为"退回"（见 Q18 收敛）✏️

- ✏️ 原三态分治（终态丢弃/处理中丢弃/挂起重处理）**收敛**为 Q18 的单一规则：**编号已存在即退回**。
- 退回单：更新原单内容（不变号）→ `status=pending_manual` → 🆕 打**退回标签** + `return_count+1`（第 N 次退回）→ 飞书通知人工 → ✏️**v2.1 修正**：照常跑 AI 流水线（结果作人工参考），末端一律转人工（原 v2 为"不跑 AI"，见《业务规则文档》第1条 FR-01）。

### Q13 · 状态枚举映射 + 两周观察期 🆕✏️

回写映射：

| 本地动作 | 回写 KSM | 回写智齿 | assistant |
|---|---|---|---|
| FAQ命中/D答复(关单) | `handleKsmOrder`→**处理完成(4)** | `save_ticket_reply`→**已解决(3)** | 飞书私信 |
| C 补充资料 | `supplyKsmOrder`(不关单) | `save_ticket_reply`→**等待回复(2)** | 飞书私信 |
| 不接管 | 无回写 | `save_ticket_reply`→**已解决(3)**(话术) | 飞书私信 |
| A bug(`pending_rd`)/B 需求(`pending_manual`) | 不回写 | 不回写 | 飞书私信 |

🆕 **两周观察期生命周期**（关闭 = 纯本地操作，到期一律不回写）：

| 阶段 | 动作 | 状态 |
|---|---|---|
| 答复关单 | 回写来源系统→**处理完成**，记 `resolved_at` | 本地仍非 closed |
| 14 天无退回 | 扫描器到期 | ✏️ **本地 `closed`（不回写来源系统）** |
| 14 天内被退回 | 取消关闭 | 转人工（Q18） |

- KSM / 智齿**都有 14 天观察期**；**assistant 无观察期**（D 答复直接 `done`）。
- 14 天可配置。来源系统**全程只回写一次**（关单那次）。

### Q14 · PII 存储 ✅

- `raw_json` 原样明文存（内网安全边界，锁定决策8）。PII 仅在回写来源系统时取用，**绝不进 LLM prompt**（出站脱敏保证）。依赖 DB 访问控制 + 备份管控。

### Q15 · 附件回写 ✏️🆕

- 附件（图片及全类型）统一存 MinIO；回写客户**一律附 MinIO 链接**，不往来源系统塞二进制。FAQ 图片、工单原始附件同存 MinIO。
- ✏️ 5M/压缩逻辑作废（不传本体则不需要）。
- ✅ **图片出口已确认**：MinIO 对外访问地址 = `https://fpy-jfsv.kingdee.com:8864/`（HTTPS，公网/Agent 可达）。回写客户图片链接 + ask agent images URL 均用此出口拼接。P4"内网直连"用于服务端读写，对外暴露走此 HTTPS 地址，二者不冲突。

### Q16 · 集成开关 ✏️🆕

- **Linear**：本期**真启用**（Team `CNPRD`，token 走环境变量）。
- **飞书通知**：统一走**一个可配置机器人 webhook**（🔗 默认值已提供，存 `FEISHU_BOT_WEBHOOKS`，**支持多个**）；所有场景共用此机器人 + 按场景模板匹配。
- **回写**：✏️ **直接真回写**，不走 DRY_RUN 灰度。

### Q17-19 · 智能助手（本期收窄为只读统计机器人）✏️

- ✏️ **本期助手只做只读 NL2SQL 统计，不做任何写/操作触**（关闭工单、改责任人、按钮操作等全部砍掉，降为后续规划）。
- 形态：自然语言指令 → 后台大模型转 SQL → 执行 → 对话机器人返回结果（可图表/表格）。**全字段、全表可统计**。
- 护栏：
  - **强制 SELECT-only**（禁 UPDATE/DELETE/DROP）。
  - **不限制表**（含 users/权限/审计表均可查）。
  - **结果 LIMIT 封顶**（防全表拖库）。
  - **PII 列（手机/邮箱/税号等）按角色脱敏**：visitor 看脱敏，handler/admin 看明文。
- 权限：admin/handler/visitor 三级均可用只读统计。

### Q18 · 退回识别 ✏️🆕

- ✏️ **退回识别规则（最简）**：**推送过来的工单编号（`source_id`）在系统已存在 = 退回**。不看状态码、不做内容 diff。
- 不分原单当前状态（处理中也算），一律转人工。处理中被退回理论不会发生，真发生也转人工。
- 无"客户在线追问/多轮对话"场景——客户不满意的唯一表现 = 工单被退回。

### Q19 · 多点故障 / 背压 🆕

- **队列积压**（待处理 >N 或 最老任务等待 >M 分钟）→ 额外发**系统级飞书告警**给运维/管理员，与按责任人 SLA 通报区分。
- **关键依赖全挂**（agent+DeepSeek 都不可用）→ 任务**挂起不消费、暂停拉取**，探活恢复后自动继续，不耗光重试变 abandoned。
- **abandoned 任务** → 飞书告警 + 后台**手动"重新入队"**口子。
- 🔗 阈值 N/M 走配置。

### Q20 · 收尾项 + ask agent 真实契约 ✏️🆕

收尾：
1. dev_owner 无映射 → 🆕 设**默认接口人**（走配置），不留空。
2. 转人工 B/C → **群发**，本期不做 `t_dispatch_assignee` 派单。
3. 观察期 14 天可配置。
4. 🔗 队列阈值 / SLA 接收群 / FAQ 审核人名单 → 先占位、联调补。

✏️ **ask agent 契约重写**（依据 `open-api-channel.md`，修正原基线 S6）：

| 项 | 原基线假设 | 实际契约 |
|---|---|---|
| 端点 | `POST /zhichi/ask {question,cid}` | 四步：`get_token`→`ask_init`→`answer_async`→`end_session` |
| cid | "固定值后续提供" | ✏️ **`ask_init` 每会话动态生成**（非固定配置） |
| 鉴权 | — | `md5(appid+create_time+app_key)`→token(24h)，同智齿签名 |
| 调用 | 同步 | ✏️ **异步 `answer_async` + 轮询**（worker 队列模型契合） |
| 会话 | — | 🆕 **每工单一会话**（ask_init→answer→end_session），规避同 cid 独占并发 |
| 路由信号 | 首行关键词 | 🆕 **先看 `transfer_result`**：TRANSFER→C；NO_ACTION→语义分类 |
| 图片 | — | `images[]`（公网可达 URL，≤5 张，≤5MB，多模态） |
| 异常 | 失败→C | 轮询 ERROR/超时/100002 → 转 C；token 失效刷新重试 |

## 七、去 AI 化环节（reply-humanize）🆕

| 项 | 决策 |
|---|---|
| 作用范围 | **仅 D 正常答复回写客户**（FAQ 命中=历史成稿、C 补料/不接管=配置话术，均不过） |
| 顺序 | agent 原文 → 提取两段(问题根因+解决方案) → **reply-humanize(去AI化)** → **desensitize(脱敏)** → writeback。脱敏放最后防破坏占位符 |
| 实现 | LLM 可热改 SKILL，提示词"改写成自然人工客服口吻、去 AI 腔/免责声明、保留事实" |
| 红线（SKILL.md 硬约束） | 只改表达，**不改事实/解决步骤**（不得篡改具体操作动作） |
| 降级 | LLM 失败 → 用原文回写，不阻断关单，记 log |
| FAQ 收录取数 | faq-record 收录用 **humanize+脱敏后**版本（FAQ 库存高质量人工口吻内容） |

D 分支出站链路：
```
D 正常回答 → 提取两段 → reply-humanize(去AI化) → desensitize(脱敏) → writeback 关单
                       → faq-record(用humanize后版本归纳) → faq-review 审核
```
